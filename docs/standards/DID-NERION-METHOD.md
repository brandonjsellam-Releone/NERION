# did:nerion DID Method Specification — Draft Outline

**Status:** Draft — Not yet registered with W3C DID Method Registry
**Version:** 0.1.0
**Date:** 2026-06-24
**Authors:** Nerion Protocol Working Group / TRELYAN
**Repository:** https://github.com/brandonjsellam-Releone/NERION

> **Additive constraint (B12 Phase-A):** This specification is a pure presentation
> and interoperability layer. It does NOT alter Nerion wire formats, SuiteIDs,
> PermitToken MAC structure, KAT vectors, or any CNSA cryptographic primitives.
> The W3C-VC and DID layers are projection only — all signing and verification
> logic remains unchanged in the Nerion core.

---

## 1. Overview

`did:nerion` is a W3C Decentralized Identifier (DID) method for identifying
**governance authorities** in the Nerion protocol. A `did:nerion` DID names the
entity that holds authorization root over an AI agent or agent swarm — not the
agent's runtime identity, but its **governance root**.

This distinction is deliberate. Nerion governs the *verb* (the action), not the
*eye* (the observer). `did:nerion` anchors who is authorized to govern, enabling
W3C Verifiable Credential chains in which a governance authority issues
`NerionPermitCredential`s to agent subjects.

### 1.1 Design Goals

- **Quantum-resistant from day one.** Verification keys are ML-DSA-87 (FIPS 204
  / CRYSTALS-Dilithium level 5). No RSA or ECDSA in the trust root.
- **Additive to Nerion core.** No new cryptographic primitives are introduced
  into the protocol; `did:nerion` projects existing governance root certificates
  into the W3C DID Document format.
- **Interoperable.** Conformant with W3C DID Core 1.0 and W3C VC Data Model 2.0
  so that any W3C-VC–compliant system can verify Nerion-issued credentials
  without a Nerion-specific runtime.
- **Audit-friendly.** The DID Document is self-describing and verifiable against
  the Nerion Merkle governance log without requiring trust in a resolver service.

## 2. DID Syntax

```
did:nerion:<network-id>:<governance-root-hash>
```

### 2.1 Components

| Component | Description |
|---|---|
| `nerion` | The DID method name (registered string). |
| `network-id` | Identifies the Nerion governance network. MUST be one of the registered values (see §2.2) or a UUID v4 for private networks. |
| `governance-root-hash` | A 32-byte truncated SHA-512/256 hash of the ML-DSA-87 public key, encoded as `base58btc` with the multibase prefix `z`. |

### 2.2 Registered network-id Values

| Value | Description |
|---|---|
| `mainnet` | Primary Nerion public governance network. |
| `testnet` | Public test network — credentials carry no governance weight. |
| `devnet` | Ephemeral development network. |
| UUID v4 | Private / enterprise network. |

### 2.3 ABNF Grammar

```abnf
did-nerion        = "did:nerion:" network-id ":" governance-root-hash
network-id        = "mainnet" / "testnet" / "devnet" / uuid-v4
uuid-v4           = 8HEXDIG "-" 4HEXDIG "-" "4" 3HEXDIG "-" [89ab] 3HEXDIG "-" 12HEXDIG
governance-root-hash = "z" 1*( base58-char )
base58-char       = %x31-39 / %x41-48 / %x4A-4E / %x50-5A / %x61-6B / %x6D-7A
```

### 2.4 Example

```
did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H
```

## 3. DID Document

A `did:nerion` DID Document is a JSON-LD document conforming to W3C DID Core
1.0. The document is derived from a Nerion governance root certificate and MUST
NOT be manually authored — it is always generated by the Nerion resolver from the
on-chain governance root cert.

### 3.1 Normative Example

```json
{
  "@context": [
    "https://www.w3.org/ns/did/v1",
    "https://nerion.dev/contexts/did/v1"
  ],
  "id": "did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H",
  "verificationMethod": [
    {
      "id": "did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H#key-1",
      "type": "NerionMLDSA87VerificationKey2026",
      "controller": "did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H",
      "publicKeyMultibase": "z<ML-DSA-87 public key bytes, base58btc encoded>"
    }
  ],
  "assertionMethod": [
    "did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H#key-1"
  ],
  "authentication": [
    "did:nerion:mainnet:zQmW5ZE7kCYhST8HkAnV5cJnQ1iGDX7MLoD3F5aMCbq9H#key-1"
  ],
  "nerionGovernance": {
    "merkleLogEndpoint": "https://nerion.dev/log/mainnet",
    "governanceRootCertHash": "sha512-256:<hex>",
    "suiteVersion": "Ps1"
  }
}
```

### 3.2 Context Extension (`nerion.dev/contexts/did/v1`)

The Nerion DID context defines the following terms not present in the base DID
context:

| Term | Type | Description |
|---|---|---|
| `NerionMLDSA87VerificationKey2026` | `@type` | ML-DSA-87 (FIPS 204) public key for governance verification. |
| `nerionGovernance` | property | Extension object linking the DID Document to the Nerion Merkle log. |
| `merkleLogEndpoint` | property | HTTPS URL of the Nerion Merkle governance log for this network. |
| `governanceRootCertHash` | property | SHA-512/256 hash of the governance root certificate, hex-encoded. |
| `suiteVersion` | property | Nerion suite version identifier (e.g., `Ps1`). INFORMATIONAL only — does not alter verification logic. |

## 4. CRUD Operations

### 4.1 Create

A `did:nerion` DID is created by:

1. Generating an ML-DSA-87 key pair (external to Nerion core; MUST use a FIPS
   204 compliant implementation).
2. Submitting a signed **governance root certificate** to the target Nerion
   network. The certificate format is defined in the Nerion protocol spec (ADR-0019).
3. The network logs the certificate in the Nerion Merkle governance log.
4. The `governance-root-hash` component of the DID is computed as:
   `base58btc(SHA-512/256(ml_dsa_87_public_key_bytes)[0..32])`.

The DID is considered **active** once the governance root certificate reaches
log inclusion (a Merkle inclusion proof is available).

**Additive constraint:** The Create operation does not alter any existing Nerion
wire format or SuiteID. The governance root certificate is an existing Nerion
artifact; the DID is a projection of it.

### 4.2 Read (Resolution)

Resolving a `did:nerion` DID proceeds as follows:

1. Parse the `network-id` and `governance-root-hash` from the DID string.
2. Query the Nerion Merkle log for the network identified by `network-id`.
3. Locate the governance root certificate whose public key hashes to
   `governance-root-hash`.
4. Verify the Merkle inclusion proof for the certificate.
5. Construct and return the DID Document from the certificate fields.

A **universal resolver driver** (conforming to the DIF Universal Resolver
interface) will be provided at `https://nerion.dev/resolver/`.

**Offline / air-gapped resolution:** A verifier that holds a local copy of the
Merkle log can resolve and verify without contacting the resolver endpoint.

### 4.3 Update (Key Rotation)

Key rotation preserves the DID identity. To rotate:

1. The current key holder issues a **governance key rotation certificate** signed
   by the existing ML-DSA-87 key. The certificate references the new ML-DSA-87
   public key.
2. The rotation certificate is submitted to the Nerion network and logged.
3. The DID Document's `verificationMethod` is updated by the resolver to reflect
   the new key.

The DID string itself does NOT change on key rotation — only the `#key-N`
fragment index increments.

**Non-update fields:** `network-id` and `governance-root-hash` are immutable
for the lifetime of the DID. If the governance root must change, a new DID MUST
be created.

### 4.4 Deactivate (Revocation)

Revocation is non-reversible:

1. The governance authority (or a designated revocation delegate) issues a
   **governance revocation certificate** signed by the current active key.
2. The revocation certificate is submitted to the Nerion network and logged.
3. The resolver returns a DID Document with `"deactivated": true` and removes
   all verification methods from `assertionMethod`.

There is no "un-revoke" operation. A new DID MUST be created if governance is
to be re-established.

## 5. Security Considerations

### 5.1 Quantum Resistance

All `did:nerion` verification keys are ML-DSA-87 (FIPS 204, security level 5,
Dilithium-based). This provides at minimum 256-bit post-quantum security against
Grover and Shor class attacks. RSA and elliptic-curve keys are not supported in
`did:nerion` DID Documents.

> **Disclaimer:** This specification references FIPS 204 for its cryptographic
> basis. Compliance claims (FIPS-validated, CNSA Suite B, etc.) require formal
> validation by an accredited laboratory and are not made here.

### 5.2 DID Document Integrity

The DID Document is derived deterministically from the governance root
certificate in the Nerion Merkle log. A verifier SHOULD always re-derive the
Document from the log rather than caching resolver responses, unless the
resolver response includes a valid Merkle inclusion proof.

### 5.3 Key Rotation Chain-of-Custody

Each key rotation certificate is signed by the previous active key and logged
in the Merkle log, creating an auditable chain of custody. Verifiers can
reconstruct the full rotation history from the log.

### 5.4 Resolver Trust

The Nerion resolver at `https://nerion.dev/resolver/` is a convenience
endpoint. It MUST NOT be trusted in isolation for high-assurance use cases.
Verifiers in high-assurance contexts MUST verify the Merkle inclusion proof
returned by the resolver, or resolve offline from a local log replica.

### 5.5 Privacy Considerations

- The `governance-root-hash` is a one-way hash of the ML-DSA-87 public key and
  does not by itself reveal the key or the identity of the governance authority.
- The Nerion Merkle log is append-only and public. Once a governance root
  certificate is logged, it is permanently visible in the log, even after
  revocation. Controllers should be aware of this before registering a DID on a
  public network.
- Personally identifiable information MUST NOT be included in governance root
  certificates or DID Documents.

### 5.6 Denial-of-Service

The Nerion network MUST implement rate limiting on governance root certificate
submission to prevent log flooding attacks.

### 5.7 Additive Purity (B12 Phase-A)

The `did:nerion` DID method is a projection layer. It reads from and presents
Nerion governance artifacts but does not write to, alter, or extend the Nerion
signing or verification logic. All CNSA primitives, wire tags, SuiteIDs, and
KAT vectors remain untouched. The gate suite MUST remain green after any
implementation of this specification.

## 6. Relationship to Nerion Protocol Components

### 6.1 ADR-0019 (B12 Phase-A: W3C-VC Projection)

`did:nerion` is the identifier layer that makes ADR-0019's W3C-VC projection
coherent. The full W3C-VC chain enabled by this method:

| Role | DID | Artifact |
|---|---|---|
| **Issuer** | `did:nerion:<governance-authority>` | Issues `NerionPermitCredential` |
| **Subject** | `did:nerion:<agent-identity>` or any W3C DID | Receives credential |
| **Credential** | — | `NerionPermitCredential` (from `vc-projection.ts`) |

The `vc-projection.ts` module wraps an existing Nerion PermitToken into a W3C
Verifiable Credential. It does not re-sign or alter the PermitToken's MAC
structure; the VC envelope is additive metadata only.

### 6.2 ML-DSA-87 in Nerion Core

ML-DSA-87 is already the signing algorithm for Nerion governance root
certificates (CNSA 2.0 alignment). `did:nerion` reuses this existing key
material; no new key generation ceremony is required for DID creation beyond
what Nerion governance already requires.

### 6.3 Merkle Governance Log

The Nerion Merkle log (ADR-0005, quorum receipts) serves as the authoritative
registry for `did:nerion` identifiers. The log's append-only, Merkle-auditable
structure provides the tamper-evidence property required by W3C DID Core §8.2.

## 7. W3C DID Method Registry Path

### 7.1 Registration Requirements

To register `did:nerion` in the W3C DID Method Registry
(https://www.w3.org/TR/did-extensions-methods/), the following are required:

1. **Specification at stable URL.** This document, once finalized, will be
   published at `https://nerion.dev/specs/did-nerion/` with a stable version
   anchor.
2. **Basic resolution implementation.** A working resolver endpoint and a
   reference implementation of the resolver algorithm.
3. **Contact information.** TRELYAN as the responsible party.

### 7.2 Prerequisites

Registration will not be pursued until:

- External security audit is complete and findings are addressed.
- FTO (Freedom to Operate) review covering `did:nerion` identifier scheme and
  W3C-VC integration has been conducted. **No FTO has been conducted as of this
  draft. No non-infringement claim is made.**
- The Nerion Merkle governance log has a public testnet deployment.

### 7.3 Estimated Timeline

6–12 months from this draft, contingent on audit completion and FTO clearance.

## 8. Strategic Context

### 8.1 Standards Positioning

A registered W3C DID method places Nerion in the same interoperability tier as
`did:key`, `did:web`, and `did:ion`. Every W3C-VC compliant system — including
federal identity infrastructure — can recognize and verify Nerion governance
credentials without a Nerion-specific runtime.

### 8.2 Federal SSI Procurement

U.S. federal agencies are increasingly requiring W3C DID and VC compliance for
identity and authorization systems (see NIST SP 800-63C, OMB Memorandum M-22-09).
A registered `did:nerion` method enables Nerion to participate in federal
Self-Sovereign Identity (SSI) procurement conversations and FISMA-aligned
deployments.

### 8.3 Differentiation from SIGA

`did:nerion` governs **the verb** (what the agent is authorized to do) rather
than the observer layer. This is the same design-around principle that
differentiates Nerion's governance model from SIGA's Commit-Point Gate patent
claim. The DID method specification makes this distinction explicit and
auditable at the standards layer.

## 9. Implementation Notes

### 9.1 Reference Implementation

A reference implementation of the `did:nerion` resolver will be provided as a
TypeScript module in the Nerion repository at
`src/did/nerion-resolver.ts` (not yet implemented as of this draft).

### 9.2 Multibase and Multicodec

Public keys in `did:nerion` DID Documents use the `multibase` encoding with the
`base58btc` alphabet (prefix `z`). The ML-DSA-87 key type will be registered
with the `multicodec` table once a codec identifier is assigned by the
multicodec maintainers.

### 9.3 Test Vectors

Known-Answer Test (KAT) vectors for `did:nerion` DID creation, resolution, key
rotation, and revocation will be published in `tests/kat/did-nerion/` as part
of the B12 Phase-A test suite. These vectors MUST NOT alter any existing KAT
vectors for the Nerion core protocol.

## 10. Open Questions

The following questions are deferred for the next draft revision:

1. **Governance root cert format normalization.** Should the resolver normalize
   governance root cert fields into a canonical JSON form before hashing, or use
   raw bytes? (Impacts interoperability with non-TypeScript implementations.)
2. **DID deactivation observability.** Should the Nerion log emit a separate
   "revoked" event, or is checking for a revocation certificate sufficient?
3. **Multi-key DID Documents.** Should a single governance authority support
   multiple simultaneous active keys (e.g., for geographic redundancy)?
4. **Multicodec registration.** What codec byte prefix should ML-DSA-87 use in
   `publicKeyMultibase` encoding? Coordination with the multicodec community is
   needed.
5. **DIF Universal Resolver driver.** Timeline and ownership for contributing a
   `did:nerion` driver to the DIF Universal Resolver.

## Appendix A — Glossary

| Term | Definition |
|---|---|
| DID | Decentralized Identifier. W3C standard (https://www.w3.org/TR/did-core/). |
| DID Document | JSON-LD document describing a DID subject's public keys and service endpoints. |
| Governance Authority | The entity holding signing authority over a Nerion governance root certificate. |
| Governance Root Certificate | A Nerion protocol artifact binding a governance authority's ML-DSA-87 public key to a set of governance permissions. |
| ML-DSA-87 | Module-Lattice Digital Signature Algorithm, NIST FIPS 204, security level 5 (equivalent to AES-256). |
| Merkle Log | Append-only, Merkle-tree-structured log used by Nerion to record governance events with tamper-evidence. |
| NerionPermitCredential | A W3C Verifiable Credential wrapping a Nerion PermitToken. Defined in ADR-0019 / `vc-projection.ts`. |
| PermitToken | A Nerion protocol artifact authorizing a specific AI action, carrying a MAC over the intent commitment. |
| SSI | Self-Sovereign Identity. Identity model in which the subject controls their own identifiers and credentials. |
| W3C-VC | W3C Verifiable Credential. Standard for cryptographically verifiable claims (https://www.w3.org/TR/vc-data-model-2.0/). |
