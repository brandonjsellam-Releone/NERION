# Nerion — Extended Assurance Matrix

> **Reading contract.** This document deepens the A39 guarantee×artifact matrix in
> [ASSURANCE.md](ASSURANCE.md). Where ASSURANCE.md answers *"how true is the claim?"* at the
> product level, this file answers *"for each specific security guarantee, exactly what artifact
> delivers it, how was it verified, and what remains open?"* at the engineering level. Every row
> carries an honest **Status** and an explicit **Open Gaps** column. Nothing in this document
> overrides or supersedes ASSURANCE.md — they are complementary views of the same evidence base.
>
> **Boilerplate that applies to every row without exception:**
> - "Tested" means specific example cases pass. It is not a proof and not fuzzing.
> - "Property-checked" means a randomized invariant survived a generated input space. It is
>   evidence, not a proof.
> - "Conformance-checked (Cn)" means the implementation agrees with Nerion's own pinned vectors.
>   **Conformant is not validated.**
> - "UNAUDITED" means no independent external review exists. Applies to all bespoke constructions.
> - "Pre-FTO" means the engineering design-around has not received a legal freedom-to-operate
>   opinion. It is not a non-infringement claim.
> - Nothing in this document claims FIPS validation, external audit, or patent clearance.

### 2. Fail-Closed Behavior

| Guarantee | Artifact | Verification Method | Status | Open Gaps |
|---|---|---|---|---|
| **`decide()` returns `deny` on any unhandled exception** — a policy that cannot be canonicalized, a resolver that throws, or any other unexpected condition produces a tier-3 deny, never a silent allow. | `kernel/src/kernel.ts` `decide()` / `decideWithAuthorizer()` — the outer `try/catch` block emits `effect: 'deny', tier: 3, reasons: ['safe-fallback deny: …']` on any throw. ### 3. Tamper-Evident Logging (Merkle Inclusion Proofs)

| Guarantee | Artifact | Verification Method | Status | Open Gaps |
|---|---|---|---|---|
| **Every logged decision is provably included in an RFC 6962-style Merkle log** — a forged or omitted leaf is detectable by any holder of a gossiped Signed Tree Head (STH). | `translog/src/merkle.ts` — SHA3-based RFC 6962 Merkle tree with `0x00`/`0x01` leaf/node domain separation; `translog/src/log.ts` `proveInclusion` / `checkInclusion`; `translog/src/sth.ts` STH signing. | `translog/test/log.test.ts` — inclusion + tamper rejection; `merkle-soundness.test.ts` — forged-root rejected for all `0 < m < n`; conformance **C10**. | **Tested · conformance-checked (C10)** | Single-operator log unless externally gossiped — split-view is *detectable*, not *prevented*. There is no production gossip network; verification rests on a client holding an independently-obtained STH. |
| **Tampered or substituted leaves are rejected** — altering any leaf byte changes the root and fails the inclusion proof. | `translog/test/merkle-soundness.test.ts` — exhaustive loop over all forged-leaf positions for a range of tree sizes. | **Tested (near-exhaustive for small trees)** | The soundness test covers small-to-medium trees exhaustively; it is not a cryptographic proof. Collision resistance is inherited from SHA3-256 (classical; NIST approved). |
| **Consistency (append-only) proofs detect history rewrite** — `checkConsistency` verifies the old root was a prefix of the new root, so an operator cannot silently rewrite past entries. | `translog/src/merkle.ts` `verifyConsistency` — TLOG-001 fix: size-0 consistency requires matching the empty-tree root, not an arbitrary root; TLOG-002: verification is capped at `MAX_TREE_SIZE = 2³¹−1` (32-bit-safe bitwise ops); TLOG-DOS-001 (round-2): proof length bounded to `2·bitLength(n)+1` before the hash chain. | `translog/test/log.test.ts` — append-only consistency tests; tamper-rejection. | **Tested · conformance-checked (C10)** | Verification is 32-bit-safe only (`MAX_TREE_SIZE = 2³¹−1`). Lifting this cap requires BigInt index arithmetic (tracked but not yet implemented). Proof-length bound is a DoS guard; soundness is unchanged by surplus elements (root comparison still fails them). |
| **Log leaf hiding — intent commitment is salted** (RCPT-001 / ADR-0014) — the `commitments.intent` field in the public leaf is a salted SHA3 hash, so a low-entropy field (e.g. `amount`) cannot be brute-forced by a log observer. | `receipts/src/receipt.ts` — `INTENT_SALT_BYTES = 32`; `commitField(intent, salt)` from `disclosure/src/selective.ts`; the salt lives on the `Receipt` wrapper, never in `ReceiptBody` / the leaf. Also `inputHash` and `decisionHash` are salted (RCPT-002: raw replay hashes transitively leak amount via the embedded input). | Conformance **C23** asserts: salt-less observer cannot recompute; authorized verifier (with salt) can; wrong salt / tampered amount is rejected. | **Tested · conformance-checked (C23)** | Hiding is computational / ROM — it rests on the 256-bit salt staying secret to unauthorized parties. Not information-theoretically hiding. UNAUDITED. Full per-field salting (granular selective disclosure without sibling revelation) is deferred. |

### 4. Quorum Security (2/3 Threshold, BigInt Cross-Multiply)

| Guarantee | Artifact | Verification Method | Status | Open Gaps |
|---|---|---|---|---|
| **A quorum receipt finalizes if and only if ≥ k distinct, valid, member ML-DSA-87 signatures are present** — duplicated or non-member signatures cannot inflate the count. | `receipts/src/quorum.ts` `verifyQuorumReceipt()` — `seen` set deduplicated by validator pubkey; membership check against `set.validators`; `k < 1` fails closed (the k=0 trivial-accept bug). | `receipts/test/quorum.property.test.ts` — randomized property: "finalizes IFF ≥ k distinct valid MEMBER sigs" over generated signer sets, plus set-substitution rejection; conformance **C12**. | **Property-checked · conformance-checked (C12)** | Proves a quorum *signed*, not that signers are Sybil-resistant or the action is safe. Key distribution and validator-set registration are deployment responsibilities outside the protocol boundary. |
| **Stake-weighted quorum uses BigInt arithmetic** — floating-point accumulation past 2⁵³ would round a sub-threshold coalition's stake up to the threshold, silently passing. Fixed: all stake accumulation is BigInt (LEDGER-PRECISION-001/003, ADR-0027). | `receipts/src/quorum.ts` `verifyQuorumReceiptByStake()` — `stake` accumulates as `bigint`; `stakeThreshold` must be `bigint > 0n`; runtime type guard rejects a non-bigint stake with a `false` verdict (not a throw). `ledger/src/sortition.ts` `safeStake()` / `isWellFormedStakeSet()`. | `receipts/test/quorum.property.test.ts` — stake-weighted variants; ADR-0027 regression test for runtime non-bigint stake (rejected without throwing); conformance C12 / C23 unchanged (dCBOR encodes same-value number/bigint identically). | **Property-checked · tested** | BigInt accumulation is wire-compatible with all ≤2⁵³ stakes (dCBOR encoding is identical). Stakes >2⁵³ are a new capability; no conformance vector currently exercises that range. |
| **Validator-set substitution is rejected** — `verifyQuorumReceipt` recomputes the `setId` from the verifier's own trusted `ValidatorSet` and rejects any receipt whose committed `setId` does not match (the load-bearing multi-model council finding). | `receipts/src/quorum.ts` `quorumSetId()` — SHA3 over canonical CBOR of sorted `[pubkey, stake]` list + k + epoch + context tag; `verifyQuorumReceipt` compares recomputed vs. committed `setId`. | Property test + Team Apex 2026-06-21 adversarial review confirmed substitution-rejection holds. | **Property-checked · conformance-checked (C12)** | Does not prevent a verifier that is fed a permissive set *as its own trusted set* from being deceived — that is a key-management / trust-anchor question, not a protocol one. |
| **DOS-VERIFY-001 — per-member PQ verification capped at one attempt** — without the `attempted` guard, duplicate garbage-sig attestations for a known member each ran a fresh ML-DSA-87 verify, giving an attacker O(N) unbounded CPU. Fixed: `attempted` set prevents a second verify per distinct member. | `receipts/src/quorum.ts` `countDistinctValid()` — `attempted.has(a.validator)` check added (round-2 sweep); `seen` only set on a valid sig; `attempted` set on first try regardless of outcome. | Code-review finding (Team Apex round-2); covered by existing property tests (all distinct members, no duplicate-attestation cases explicitly tested beyond the count invariant). | **Tested (incidentally)** | No explicit adversarial stress-test for a crafted receipt with N×duplicate attestations. An explicit DoS-profile test would close this gap. |
| **Accountable safety is formally modeled** — a TLC-checkable TLA⁺ model encodes stake-finality + Casper-style accountable safety: two distinct blocks finalized at one height ⇒ ≥1/3 stake equivocated (slashable); honest validators are never slashable; <1/3 Byzantine stake ⇒ agreement. | `docs/formal/NerionConsensus.tla` (+ `.cfg`, README) — faithful to `ledger/src/equivocation.ts` and `governance/src/quorum.ts`. Invariants `AccountableSafety`, `HonestAgreement`, `NoHonestEquivocation`, `QuorumIntegrity`. | TLA⁺ model checking (TLC) over a small finite configuration (4 validators). | **Modeled (not yet TLC-run-in-CI)** | A model of an **abstraction** — model checking finds counterexamples; it does **not** prove the TS/Rust implementation correct, and abstracts away signatures, networking, timing, and view-change. TLC is not yet wired into CI. |

### 5. Key Isolation (Per-Plane Distinct Key Material)

| Guarantee | Artifact | Verification Method | Status | Open Gaps |
|---|---|---|---|---|

### 8. Replay Protection (Sequence Numbers, Epoch Binding)

| Guarantee | Artifact | Verification Method | Status | Open Gaps |
|---|---|---|---|---|
| **PermitTokens carry a tight expiry and are bound to a specific action hash** — a stolen permit cannot be replayed for a different action, a different audience, or beyond its `exp` field. | `planes/src/permit.ts` — `PermitClaims` includes `exp`, `actionHash`, `audience`, `sessionId`, `effect`; `verifyPermitForAction` fails closed on non-finite `exp` (PS-PLANE-05); `actionHash` is `SHA3(canonical(intent))`. | `planes/test/` permit tests; conformance C22 (audience binding). | **Tested · conformance-checked (C22)** | True single-use within the expiry window is the **resource's idempotency responsibility** (M-P1-1 in the threat model). The protocol binds the token to one action but does not prevent a resource from honoring it twice within the window. No distributed sequence number / nonce-history store on Plane 1 (by design — latency budget). |
| **Quorum receipts are epoch-bound** — a receipt committed to epoch N is rejected by a verifier at epoch M≠N; changing the validator set or threshold increments the epoch and creates a different `setId`. | `receipts/src/quorum.ts` `quorumSetId()` includes `epoch`; `verifyQuorumReceipt` checks `q.epoch !== epoch`. | `receipts/test/quorum.property.test.ts`; conformance C12. | **Property-checked · conformance-checked (C12)** | Epoch management (incrementing on set rotation) is a deployment obligation. The protocol enforces epoch binding but has no autonomous epoch-rotation trigger. |
| **Settlement credit grants are one-shot via (account, nonce) binding** (SETTLE-001) — a signed `CreditGrant` cannot be replayed to double-credit: the `nonce` is consumed on first use. | `settlement/src/credits.ts` — `(account, nonce)` uniqueness check; SETTLE-001 fix, Team Apex 2026-06-21. | `settlement/test/` — regression for replay attempt; Team Apex finding. | **Tested** | Nonce uniqueness relies on a per-account nonce store; if that store is reset or shared across shards without coordination, replay becomes possible. Distributed nonce management is a deployment constraint. |
| **View-change certificates are bound to (height, prevHash, round)** — a cert cannot be replayed across forks or rounds (LEDGER-VRF-001 / sub-⅓ grind fix). | `ledger/src/chain.ts` — `viewChangeCert` binding includes `(height, prevHash, round)`; non-negative-integer round enforced. | `ledger/test/` consensus tests; Team Apex 2026-06-21 liveness sweep. | **Tested** | The `≥⅔` round-skip re-draw (LEDGER-007) is a documented fairness-only weakness: a coalition controlling liveness can manipulate leader selection across rounds; chain safety is unaffected. Rigorous fix (round-0 cert chain) is roadmapped. |
| **`replay()` hash is over the raw `inputBytes`, not the decoded value** — ensures one canonical CBOR byte string corresponds to exactly one receipt hash, preventing parser-differential receipt multiplication. | `kernel/src/replay.ts` — `inputHash = SHA3(bundle.inputBytes)` before decode; `decodeCanonical` enforced for decide. | `kernel/test/` replay tests. | **Tested** | REPLAY-CANON-001 is closed in the reference implementation. A non-reference actuator that builds bundles from a lenient CBOR encoder (e.g. one that emits duplicate keys) would produce bundles that the reference replay rejects — closing the attack by failing closed, not by preventing the malformed input. |

## Cross-Cutting Gaps (honest summary)

The following gaps are present across multiple categories and not fully addressed by any single row above.
