# Team Apex — Round-3 audit (2026-06-21): novel surfaces

**Scope:** attack-classes/modules not yet deeply audited — settlement, ops/env-handling, CBOR decode-DoS,
constant-time/side-channels, keystore custody providers (KMS/HSM), cross-module composition, hybrid-KEM /
session-key establishment, and the v:2 disclosure composition. 8 surfaces, CHUNK=2 throttle (no rate-limit
this round, all 8 ran), every candidate dual-refuter verified. As in Round 2b, finder *summaries* were
re-checked by hand for concrete candidates the automated gate dropped.

## Fixed in this session (gate green)

### LEDGER-PRECISION-002 — view-change quorum check uses IEEE-754 `number` (MEDIUM)
`ledger/src/leader.ts:106` (`verifyViewChangeCert`). The sibling fix LEDGER-PRECISION-001 converted the
finality cross-multiply (`chain.ts`), the sortition walk (`sortition.ts`), and `vrfLeaderEligible`
(`leader.ts:39`) to BigInt — but **missed this fourth quorum cross-multiply**. `stake`/`total` are unbounded
PoS weights; past 2^53 the products lose precision and `stake*finalityDen >= finalityNum*total` can flip:
a sub-2/3 view-change cert accepted (a SUB-1/3-coalition round-advance / VRF re-draw — stronger than the
disclosed LEDGER-007), or a legit ≥2/3 cert rejected (liveness). It does **not** break final-block safety
(the block's own finality check is BigInt-hardened). **Fix:** `BigInt(stake) * BigInt(finalityDen) >=
BigInt(finalityNum) * BigInt(total)`, parity with `chain.ts`.

### SETTLE-METER-001 — `meter()` has no replay/idempotency on `ref` (MEDIUM)
`settlement/src/credits.ts`. `grant()` got replay protection (SETTLE-001, `consumedNonces`), but the
`meter()` analogue had none — though `ref` exists precisely to identify the metered decision/receipt. A
replayed Plane-1 permit (threat R2) drives `meter(acct, cost, sameRef)` repeatedly, each decrement draining
the victim's budget (griefing/DoS). **Fix:** a `meteredRefs` set keyed on `(account, ref)`; a repeat is
rejected. Marked only after the balance check, so a failed (insufficient) meter stays retryable. Regression
tests: a reused ref cannot double-charge; distinct refs/accounts meter independently; a failed meter doesn't
consume the ref.

## Sound, zero findings
- **ops/env:** secrets never logged (Azure/AWS error helpers extract only non-sensitive ids; `.env` gitignored
  + untracked); no fail-OPEN on missing config (constructors hard-throw on empty required vars); no env value
  flows into a signed/MAC'd transcript.
- **CBOR decode-DoS:** cbor2 enforces `maxDepth:1024`, grows containers by actual `push()` (no count
  pre-allocation), validates string lengths against the buffer — the classic decode-bomb vectors are all
  mitigated at the library default `decodeCbor` inherits.
- **constant-time:** `constantTimeEqual` is the comparator for every in-trust-model secret/auth value (permit
  HMAC, COSE header, sealed-seed roundtrip, attester-trust-set, commit-binding). The non-CT comparators
  operate only on PUBLIC values (Merkle hashes, log roots, public commitments) — no secret-dependent timing
  oracle.
- **hybrid KEM / key establishment:** Nerion never assembles its own combiner (only audited noble X-Wing /
  QSF, which bind both legs + pubkeys + ciphertext); and the KEM sharedSecret has **no production consumer** —
  every `sessionKey` is an externally-supplied pre-shared secret, never KEM-derived. No KEM→sessionKey path to
  attack. The path that exists (`sessionKey → deriveAudiencePermitKey`) is correct HKDF + suite-bound MAC.

## Recorded as residual hardening (latent / non-exploitable — not fixed this round)
- **DISCLOSURE-SLOT-001 (MED, latent):** `commitField` applies one `{domain, salt}` to all three receipt slots
  (`intent`/`inputHash`/`decisionHash`) with no per-slot discriminant; an opening isn't bound to WHICH slot.
  NOT exploitable today (the three values never coincide), but binding each commitment to its slot is best
  practice (every other digest in the repo carries a context tag). Deferred to avoid a clobber with the hot
  concurrent RCPT-001/002 area + conformance C23; needs a coordinated `commitField(value, salt, context)`
  change with test/conformance regen.
- **CUSTODY-AWS-AAD-001 (MED, latent):** the AWS KMS sealer binds no `EncryptionContext` (AAD), so a valid
  ciphertext could be cross-context/cross-id swapped. Real KMS best-practice gap, but fixing it changes the
  Encrypt/Decrypt request body and would invalidate the pinned SigV4 full-request KAT (a frozen signature that
  must be regenerated by running the signer). Deferred; low blast radius (storage-write + same-key access
  required). Secondary: `load()` doesn't length-check the unwrapped seed / `sealed.sigId` before keygen.
- **SETTLE-002 (latent):** `verifyGrant` carries no replay tracking, but no code path credits based on it
  (`grant()` applies the balance directly and has `consumedNonces`), so there is nothing to double-credit
  today — a design caveat for a future redemption flow, already documented in the verifyGrant docstring.
- **COSE-ALG-001 (latent):** `alg` is hardcoded to ML-DSA-87 and decoupled from `suite`; correct for every
  active suite, non-exploitable; derive it from the suite's sig scheme when a non-ML-DSA suite is added.
